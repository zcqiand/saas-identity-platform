# Executing Plans（逐任务子代理 + 两段审查）

长构建的头号敌人是上下文腐烂：同一个会话里堆了十个任务，模型开始串味、忘约束、
改串范围。对治办法：**每个任务用一个只拿到「该任务 + 相关测试 + 约定」的新鲜子代理**，
干完立刻两段审查，再进下一个。

## 前置

- 必须已有 `docs/plans/PLAN-<REQ号>.md`（见 `writing-plans`），每个任务绑了 fn-ID、
  写死了文件路径和验证命令。
- 计划里的新 fn-ID 都已由人批准登记（`/tree-change`）。没批的不许开工。

## 每个任务的循环

### 1. 派新鲜子代理
只给它三样：**这一个任务的描述**、它要碰的**测试文件**、项目的 `CLAUDE.md` + 相关
`docs/conventions/`。不要把整个计划或前面任务的历史灌给它——那正是腐烂来源。

### 2. 子代理干活，走 red-first
先让该任务的测试红（若还没红），再最小实现转绿。不越界改计划外的文件。

### 3. 第一段审查——**计划符合性**（主线自己做）
- 改的文件是不是计划里写死的那些？有没有顺手改了别的？
- 测试是否先于实现存在、且挂了对的 fn-ID？（`red-first` 的顺序纪律在这里把关）
- 面向用户的子项，触发元素挂 data-fn 了吗？（`reachability`：没入口 = 用户点不到，L5 会红）
- 做的是不是这一个任务，没有夹带？
- 不符 → 打回子代理重做，别放行。

### 4. 第二段审查——**代码质量**（`code-reviewer` agent）
调 `code-reviewer` agent（它读 git diff、对照 CLAUDE.md 禁止事项与 conventions，
输出分级问题清单）。**阻断项必须清零**才算过，不清零不进下一个任务。

### 5. 过门
`python scripts/gate.py -p <项目>` 绿，才收这个任务。

## 并行

计划标了「可并行」的任务，可同时派多个子代理。但**多个子代理在同一工作树上改会互相踩**，
所以并行前先用 `using-worktrees` 给每条并行线开一个隔离 worktree，各自走完两段审查 + 过门
再合流。合流后再跑一次整体 gate。

## 边界

- 两段审查缺一不可：第一段查**对不对（符不符合计划）**，第二段查**好不好（质量）**。
  只做质量审查会放过「实现得很漂亮但根本不是这个任务」。
- 子代理是执行者不是决策者：它遇到计划没覆盖的判断（要不要改设计、要不要动别的 fn-ID），
  停下上报，走 `alignment` / `systematic-debugging`，不许自作主张扩范围。
