# Red First（测试先红，再写实现）

本 suite 的 `trace.json` 把 fn-ID 映射到覆盖它的测试，L4/L5 事后核对覆盖。
但「覆盖」不等于「测试驱动」——事后补的测试往往只是给现有实现背书。
这个 skill 补上**顺序纪律**：实现之前，先有一个**为它而写、且此刻是红的**测试。

## 铁律

1. **没有红测试，不写实现。** 想直接写 impl？停。先去写会失败的测试。
2. **测试必须挂对 fn-ID。** 用 `@fn M01.F01.I03` 锚点标记（trace_cmd 靠它建映射）。
   挂错或不挂 = 这条覆盖不算数。
3. **一次只推进一个红。** 红 → 最小实现转绿 → 重构（保持绿）→ 下一个。
   不要一口气写十个测试再一起实现。
4. **禁止给 skip 的测试挂 fn-ID** 来伪造覆盖（CLAUDE.md 已列为硬约束，L5 会查）。

## 循环（red → green → refactor）

1. **Red**：写 `tests/xxx.test.ts`，断言目标行为，挂 `@fn <ID>`。
   跑 `python scripts/redcheck.py -p <项目> xxx` —— 它断言这个测试**此刻是红的**，
   绿了或没匹配到就 exit 1（test-first 被违反）。这是给「测试先行」上的半颗机器牙。
2. **Green**：写**最小**实现让它转绿。不多写一行「以后可能用得上」的代码。
3. **Refactor**：绿灯下清理，测试保持绿。
4. 跑 `python scripts/gate.py -p <项目>`——L4 绿、且 `trace.json` 新增了这条 fn-ID→测试映射。

## 和契约的接口

- 你写的 `@fn` 锚点 → `trace_cmd`（`TRACE_MAP=1 vitest run`）→ `.state/trace.json`。
- 计划里每个任务的「测试先行」栏（见 `writing-plans`）就是在要求这个循环。
- 想把「先红」变成机器门是难的（事后难证明曾经红过）。所以它靠纪律 + review 把关；
  `executing-plans` 的第一段审查会检查「这个任务的测试是否先于实现存在并挂了 fn-ID」。

## 边界

- Bug 修复同样先红：先写一个复现 bug 的失败测试，再修。见 `systematic-debugging`。
- 纯配置、纯样式、一次性脚本可豁免——但那是你显式判的，不是默认跳过。
