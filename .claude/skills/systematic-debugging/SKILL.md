# Systematic Debugging（根因之前，禁止修）

模型调 bug 的默认毛病：看到红就改一处试试，绿了就走——修的是症状，留的是隐患，
或者压根没修对、只是把红移到了别处。这个 skill 强制**先理解，再动手**。

## 铁律

**在你能用一句话说清「为什么会红」之前，不许改任何代码。** 乱试是禁止的。

## 四阶段

### 1. 复现与定位
- 用**确定能触发**的最小步骤复现。偶发的先想办法稳定复现，否则你在猜。
- 缩小范围：哪个 fn-ID / 哪个文件 / 哪一层。gate 红的话看清是 L1..L5 哪一道、报什么。
- **写一个复现该 bug 的失败测试**（挂对 fn-ID）。这既是定位，也是你修完的验收（见 `red-first`）。

### 2. 假设根因
- 写下**一句话**假设：「因为 X，所以 Y」。
- 找证据支持或推翻它——读代码、加日志、看 diff。**别急着改。**
- 想不出机制、只有「可能是……」，回第 1 步，范围还不够小。

### 3. 验证假设
- 用最小实验验证假设为真（比如临时打印、断点），确认 X 确实导致 Y。
- 假设错了就丢掉，回第 2 步。**不要在没验证的假设上动刀。**

### 4. 修复
- 针对**根因**做最小修复。第 1 步的失败测试转绿。
- 跑 `python scripts/gate.py -p <项目>` 全绿。确认没把红移到别处。

## 升级闸：三次修不好 = 停手，走 ADR

同一个 bug **修了三次还红**，停止继续试。这不是代码问题，是理解或架构问题：

1. 停手，别再改。
2. 用 `adr-writer` 写一份 ADR（`docs/adr/`）：现象、试过什么、为什么都不行、你怀疑的架构层面原因。
3. **把 ADR 贴给人，交给人裁量。** 别硬钻。

## 边界

- 「理解根因」是判断，机器替不了。gate 只告诉你哪红，不告诉你为什么。
- 修 bug 也走 `red-first`：先有复现的红测试，再修。绝不「改完手动点一下看好了」就算数。
