# Brainstorming（规格没批准，不许开工）

模型的默认毛病是一听到需求就冲去写代码，写出一堆用户根本不想要的东西。
这个 skill 的职责：**在动任何 `src/` 之前，把「到底要什么」问清楚、写下来、给人确认。**

## 边界（先读这条）

- 你的产出是**规格**，不是代码，也不是 REQ 文档本身。规格谈拢了，才交给
  `alignment`（`/req`）落成 `docs/requirements/`。
- **不许在这一步 scaffold 功能、改 src/、动 function-tree。** 一行代码都不写。
- 澄清是**判断**：该问什么、哪些是真需求哪些是伪需求，靠你和人对话，不是套模板。

## 流程

1. **照抄用户原话**，别急着理解。
2. **苏格拉底式发问**，一次问 1-3 个最要命的，不要一口气 20 问：
   - 边界：这个功能**不**做什么？（划走范围比划进范围更值钱）
   - 用户与场景：谁在什么情况下用？
   - 成功长什么样：能写出「当…则…」的验收句吗？写不出说明还没想清。
   - 边缘情况：空数据 / 并发 / 权限 / 失败态怎么办？
   - 依赖与约束：碰哪些已有功能（对上 fn-ID）？有没有硬约束？
3. **提方案，给取舍。** 至少给一个「更简单的版本」——多数需求都过度设计了。
   分块展示，每块短到能读完，逐块让人确认。
4. **收敛成一页规格**：目标 / 非目标 / 验收标准（「当…则…」）/ 影响的已有 fn-ID /
   待定问题。

## 交接

规格经人**明确点头**后，才继续：

- 落文档 → `alignment` 的 `/req <项目> <规格>`
- 新功能 → 走 `/tree-change` 提案拿 fn-ID（**人批准**）
- 拆任务 → `writing-plans` 的 `/plan`

没点头就停在这里。含糊而硬开工，是返工的头号来源。
