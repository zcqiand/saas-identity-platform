# Alignment

## 唯一锚点

功能子项 ID（`M01.F01.I03`）。需求、流程、设计、测试、权限码全部指向它。
不在 `output/<项目>/docs/functions/function-tree.md` 里的 ID 是悬空引用，L5 直接红。

## 边界（先读这条）

**机器管引用完整性，人管语义对齐。**

- 机器能判：ID 存不存在、唯一不唯一、父级有没有、skip 的测试有没有冒充覆盖。这些是**事实**。
- 机器判不了：这个流程步骤该不该由这些功能支撑、这份设计对不对。这些是**判断**。

所以你不要替人做对齐判断。你的活是：把表填对、把 ID 对上、把空格子指出来。
需要判断的地方，列出选项和影响面，交给人。

## 改功能清单 = 动地基，必须先提案

`docs/functions/function-tree.md` 受 PreToolUse hook 保护。直接改它会被拒绝（exit 2）。

流程：

1. 写 `.state/tree-change.json`，`approved` 必须为 `false`。**你无权批准自己。**
2. `python scripts/tree_change.py -p <项目> --report`
   —— 机器 grep 出每个 ID 现在被哪些需求/流程/设计/测试引用
3. **把报告贴给人，然后停下。** 不要顺手改任何文件
4. 人跑 `--approve`（交互式，非交互环境一律拒绝）
5. 令牌绑定 function-tree.md 的 sha，**只授权一次修改**，写完即焚
6. 改完，把提案里 `sync_files` 声明的对齐文件一并改掉

`action` 只有三种：`新增` / `变更` / `废弃`。**没有「删除」。** 行不删，号不回收。

命令：`/tree-change <项目> <想改什么>`

## 来了一个新需求

1. `output/<项目>/docs/requirements/REQ-YYYY-NNN-<短标题>.md`，用 `_TEMPLATE.md`
2. 先填「澄清记录」。有含糊就问，不要替用户脑补
3. 「验收标准」写不出「则」的删掉，那不是标准
4. 「任务拆解」
5. 「功能影响」表：
   - **新增** → 走 `/tree-change` 提案 → 人批准 → 登记（状态=规划）拿到 ID，再回填
   - **变更** → 直接引用已有 ID
   - **删除** → 走 `/tree-change`，action=`废弃`。改状态，**不删行**
6. `docs/requirements/README.md` 台账加一行
7. `python scripts/gate.py -p <项目> --only L5`

## 开发中 / 上线时

- `规划` → `开发中`：同时补 `docs/design/design-function-map.md`。
  **填不出接口和表，就是设计没做完，不要开工。**
- `开发中` → `已上线`：L5 开始警告缺设计映射、缺测试引用、不在任何流程里。
  警告不阻断，逐条看，人裁量。

## 测试怎么挂到功能上

按栈不同，语法不同，契约相同 —— 都产出 `.state/trace.json`。

```python
@pytest.mark.fn("M01.F01.I03")            # python
def test_...
```
```ts
fnTest(["M01.F01.I03"], "超出宽限判定为迟到", () => { ... });   // react-ts
```

### 这个标记能证明什么、不能证明什么

- 能证明：**存在一个会执行的测试，声称覆盖该功能子项。**
- 不能证明：这个测试真的执行到了该功能对应的代码。

标记是人写的。有人能把 `M01.F01.I03` 挂到一个 `add(a,b)` 上，机器发现不了。
**挂 ID 是一次承诺，不是一次证明。** 矩阵里的 ✅ 不等于「真的测了」。

（skip/xfail 的测试是例外：适配器在源头把它的 ID 抹掉，契约层再复核一次。假绿在物理上不可能。）

## 看对齐结果

```bash
python scripts/checks/_alignment.py -p <项目> --matrix
```

一张 md 表：功能子项 × 需求/流程/设计/测试。**空格子不是错误，是缺口。**
