# Reachability（用户点得到，才算做完）

`trace.json` 证明「这个功能有测试」。它证明不了「用户能从界面点到这个功能」。
于是会出现最坑的一种绿灯：**六道门全绿、测试全过，可界面上根本没有入口调用它。**
逻辑写好了，没接进 UI，用户用不了。这个 skill 就是堵这个洞。

## 锚点：入口挂 data-fn

面向用户的子项（`function-tree` 里类型是 `按钮/页面/查询/报表/标签页`），
必须在代码里有一个**入口锚点**，挂在真正触发它的元素上：

```tsx
<Button data-fn="M01.F01.I01" onClick={openCreate}>新增项目</Button>
<Route path="/reports" element={<Reports />} />   {/* 路由处： // @entry M01.F02.I03 */}
```

两种写法都认：JSX/HTML 元素上 `data-fn="<子项ID>"`，或调用处注释 `// @entry <子项ID>`。
`接口` 类型不用挂——它是后端，由调用它的那个 UI 元素间接可达。

## 实现锚点：@impl（给书用，不影响 L5）

`data-fn`/`@entry` 是**入口**锚点，回答「用户点得到吗」，是 L5 硬门。
`@impl` 是**实现**锚点，回答「这个 fn-ID 的逻辑落在哪个文件」，**不是 L5 要求、不挂也不会红**。

```python
# @impl M01.F01.I02   校验规则的实现
def validate(...): ...
```

它唯一的消费者是 `export_handoff.py`：导出 handoff 时，扫 `data-fn`/`@entry`/`@impl`
组成 `source_index`（fn-ID → 终态文件），供 knowledge-suite 写书引用。
面向用户的子项靠 `data-fn` 就够进 source_index；**纯后端/`接口` 子项没有 UI 入口，
想让书引用它的实现文件，就在那个文件挂 `@impl <子项ID>`**。不打算写进书就不必挂。

## L5 怎么查（已经是硬门）

`_alignment.py`（L5）扫 `source_dirs` 里的前端文件找这些锚点，然后：

- **已上线** 的面向用户子项**没有**入口锚点 → **L5 硬失败**（exit 1）。
  语义是悬空引用：清单声称上线，界面却没入口。这跟「测试引用了不存在的 ID」同级,是事实错误。
- **开发中** 的面向用户子项没入口 → **警告**（还在建，不阻断）。
- `/matrix` 多了一列「入口」，一眼看出哪些功能还没接进界面。

所以「做完」的定义变了：**测试绿 + 有 data-fn 入口 + L5 绿**，三者缺一不算完。

## 落进工作流

- `writing-plans`：面向用户的任务，验证栏必须包含「挂 data-fn 并跑 L5」。
- `red-first`：测试转绿后，别急着收——先确认入口锚点也在了。
- `executing-plans` 第一段审查：查这个任务是否挂了 data-fn。
- 验收一个功能时，`/reach <项目>` 或 `/matrix <项目>` 看「入口」列。

## 更强的一档（可选）

data-fn 证明「代码里有个挂了这个 ID 的元素」，证明不了「这个入口真的把功能跑通了」。
要那种保证，得上 E2E（playwright 等）：一个测试**点这个 data-fn 元素**、断言功能发生。
把该 E2E 挂上 `@fn`，它同时满足 trace 覆盖和可达性。没上 E2E 之前，data-fn 是便宜的 80%。

## 边界

- 机器能判「有没有挂 ID 的入口元素」，判不了「这个入口摆得对不对、好不好用」——那是人评审。
- 别为了消警告乱挂 data-fn 到不相干元素上。挂错 ID 就是伪造可达性，等同给 skip 测试挂 fn-ID。
