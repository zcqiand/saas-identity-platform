## 本文件只写「机器判不了、但本项目已经拍板」的取舍

eslint 已强制的不写在这里。写了就是噪音，还会让人误以为其余规则也有强制力。

## 已拍板的取舍

| 场景 | 本项目怎么做 | 为什么 |
|---|---|---|
| 列表渲染超过 200 行 | `content-visibility: auto` | 收益确定，成本为零 |
| `useMemo` 包裹简单表达式 | **不要**。直接算 | memo 本身的成本高于收益 |
| 第三方分析脚本 | hydration 之后加载 | 不阻塞首屏 |
| 服务端传给客户端组件的数据 | 只传渲染需要的字段 | 序列化成本随字段数线性增长 |

## 性能改动的前置条件

**没有 profile 数据，不做性能重构。**

一次性能改动的 PR 必须回答：

1. 慢在哪？（火焰图 / React Profiler 截图）
2. 改完快了多少？（同一测量方法的前后对比）
3. 代价是什么？（可读性、bundle、复杂度）

答不出第 1 条，说明这是"我觉得它会慢"，不是性能问题。

## 规则之间会打架

`rerender-memo` 说把昂贵计算提取进 memo 组件；
`rerender-simple-expression-in-memo` 说简单表达式不要 memo。

边界在哪，规则里没写，因为它取决于你的组件树。遇到这种冲突：
**列出两种写法的成本，交给人裁量。不要自己拍。**
